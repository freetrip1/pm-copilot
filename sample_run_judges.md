## ③ PRD 草稿

# 自由职业者智能催款与应收管理工具 PRD

**一句话定位**：专为独立自由职业者打造的轻量级“开票-追踪-催款”闭环工具，解决回款遗忘与催款尴尬，让每一笔收入落袋为安。

## 背景与问题

自由职业者（如独立设计师、开发者、撰稿人）在享受工作自由的同时，普遍面临严重的应收账款管理痛点：
1. **状态黑盒导致回款延迟**：项目交付后缺乏系统记录，发票状态散落在邮件和聊天记录中，经常遗忘哪些已开票、哪些未付款，导致资金回笼被动延迟。
2. **心理障碍导致不敢催款**：面对客户逾期，因碍于情面或不知如何措辞而迟迟不愿主动催款，严重影响个人现金流。
3. **对账繁琐导致效率低下**：月底年底对账时，需跨平台手动核对发票与打款记录，耗费大量时间且易出错。

**竞品现状与差异化机会**：
当前市场方案存在明显空白。Bonsai 偏重全流程且主攻海外；FreshBooks 和 Xero 属于重型会计软件，对独立工作者过于臃肿；Zoho Invoice 缺乏针对自由职业者心理的分级话术设计；Portmoneo 偏向支出端追踪，缺乏应收催收闭环。
本期产品将避开重型会计与全流程管理，专注于**“轻量级应收账款管理”**，并重点突破**“符合本土人情社会的智能催款”**，通过分级话术和草稿确认机制，解决自由职业者“不敢催”的心理痛点。

## 目标用户与典型场景

**目标用户**：独立提供专业服务且需自行处理财务与收款的自由职业者（如独立设计师、开发者、撰稿人等）。MVP阶段优先聚焦国内本土自由职业者。

**典型场景**：
1. **开票与追踪**：项目完成并开具发票后，用户在工具中录入发票信息及客户联系方式并设定账期；系统自动记录发票状态，并在到期前提醒用户。
2. **逾期与催款**：发票逾期未付款，用户点击一键催款；工具根据逾期天数自动匹配分级话术生成催款邮件草稿，用户确认或微调后手动点击发送至客户邮箱，发送后系统自动记录催款动作与时间。
3. **对账与决策**：用户收到线下打款后，在工具中手动确认回款（需录入实际到账日期）；仪表盘清晰展示已付款、未付款、逾期发票的汇总数据与明细，辅助财务决策。

## 产品目标与成功指标

**产品目标**：
- **北极星指标**：缩短平均回款周期（从发票到期到实际回款的天数）。
  - *可控性假设*：是否回款本质由客户决定，但产品通过降低催款心理门槛和提供专业话术，能有效提高催款行为发生率和触达率，从而间接加速客户付款决策。
  - *退路指标*：若MVP证明回款周期不可控或归因困难，则将北极星退回为产品可直接驱动的**“催款功能渗透率”**（使用催款功能的逾期发票占比）。
- **支撑性体验目标**：降低管理负担（代理指标：手动对账耗时减少）、消除催款心理障碍（代理指标：催款功能使用率）。

**成功指标（MVP验收最低门槛）**：
*注：以下为MVP验收节点的最低可接受门槛，上线后4周内根据实际基线数据校准为硬阈值。*

1. **采用率**：注册首周内录入 ≥ 1张发票且完成一次状态推进（生成草稿或确认回款）的用户占比 ≥ 15%。
2. **功能渗透率**：产生逾期发票的用户中，使用“催款功能”的比例 ≥ 20%。
3. **核心价值验证（催款转化率）**：催款后7天回款转化率 = （成功发送催款后7天内确认回款的逾期发票数 / 成功发送催款的逾期发票总数）≥ 5%。
   - *归因说明*：由于MVP无法进行随机分流实验，该指标存在自选择偏差（主动催款的用户/发票本身可能具有异质性）。MVP阶段仅作为**相关性验证指标**，不作为严格因果归因的“净效应”指标。后续通过同用户历史行为对比（同一用户催款发票 vs 其未催款发票的回款速度）进行准实验评估。
4. **留存率**：核心用户的按月留存率（当月至少登录1次并产生有效操作的用户占比） ≥ 30%。*有效操作枚举：录入发票/生成草稿/发送催款/确认回款。*
5. **商业价值弱信号**：高级功能点击率 / 愿意付费调研评分（本期验证价值假设，监控付费意愿弱信号）。

## 核心功能与需求清单

### P0-核心（MVP 必需闭环）

**1. 发票与客户信息管理**
- **功能描述**：支持手动输入发票金额、账期、开票日期，及关联客户的基础信息（姓名、邮箱）。金额必须>0且最大值为 9999999999.99；邮箱需符合正则格式校验；账期不得早于开票日期；所有字段均为必填。支持对未结清发票及客户信息编辑修改；支持逻辑删除发票（软删除，需二次确认）。
- **技术规范**：数据库强制使用 Decimal(12,2) 或将金额转换为“分”存储为 BigInt，严禁使用 Float/Double。
- **验收标准 (AC)**：
  - AC1: 输入合法数据，保存成功，列表可见新发票。
  - AC2: 输入金额<=0、非法邮箱、账期早于开票日，对应字段下方展示红色错误提示，无法提交。
  - AC3: 删除发票时弹出二次确认弹窗，确认后列表不显示该发票，且仪表盘统计不包含此发票。

**2. 发票状态追踪与到期预警**
- **功能描述**：采用两个正交维度表达状态。收款状态：未收、部分收款、已结清；逾期分桶（派生值）：未到期、逾期1-6天、逾期7-29天、逾期30天+。弃用凌晨集中跑批，采用“读时计算+写回缓存”策略（UTC+8）。到期前1天及逾期首日站内信提醒；逾期分桶跃迁时再次触发站内信提醒，同一分桶仅提醒一次。
- **验收标准 (AC)**：
  - AC1: 跨过到期日刷新页面，发票逾期分桶实时更新为“逾期1-6天”，并收到站内信。
  - AC2: 发票逾期第7天0点后首次查看，分桶更新为“逾期7-29天”，并收到跃迁站内信提醒。

**3. 分级催款话术与草稿生成（邮件+短信兜底）**
- **功能描述**：话术分三级：温和提醒（1-6天）、正式催款（7-29天）、最终通知（30天+）。一键生成对应级别邮件草稿，重复生成将覆盖旧草稿（覆盖前二次确认）。部分收款时，催款金额自动计算为（应收-实收）。短信兜底：无邮箱或邮件退信时，提供短信通道（直接发送预设模板，不支持草稿编辑）。
- **前置依赖**：话术模板需经法务团队审核通过后方可上线。
- **验收标准 (AC)**：
  - AC1: 逾期1-6天生成草稿，话术为“温和提醒”；修改系统时间至逾期10天，生成草稿话术为“正式催款”。
  - AC2: 部分收款（应收1000，实收400）状态下生成草稿，草稿中催款金额显示为600。
  - AC3: 覆盖已存在的草稿时，弹出“覆盖确认”弹窗；取消则保留旧草稿。
  - AC4: 短信发送后，状态更新为“发送成功”；若短信接口报错，状态更新为“发送失败”，提示用户“短信发送失败，请检查号码或稍后重试”，并允许重新发送。

**4. 邮件发送与状态追踪**
- **功能描述**：MVP强制采用用户授权 SMTP/OAuth 集成发信（From与Reply-To均为用户真实邮箱）。用户确认草稿后手动点击“发送”，发送中按钮置灰防抖（3秒）。状态枚举：待发送、发送中、发送成功、发送失败、已退信。退信时通过站内信预警，允许再次生成草稿。发送中禁止生成新草稿；发送失败/退信时覆盖需提示。
- **异常流定义**：邮件发送超时或网络异常等无响应情况，超过60秒未收到邮件网关回调，系统自动将状态由“发送中”置为“发送失败”，并提示用户，允许重新生成草稿。
- **验收标准 (AC)**：
  - AC1: 邮件发送成功，状态变为“发送成功”；模拟网关返回退信Webhook，状态变为“已退信”并收到站内信。
  - AC2: 邮件处于“发送中”时，点击“生成催款草稿”按钮，提示“上一次发送尚未完成，请稍后”。
  - AC3: 模拟发送超时（60秒无回调），状态自动流转为“发送失败”，可重新操作。

**5. 手动确认回款/销账**
- **功能描述**：用户对未结清发票手动录入实收金额与**实际到账日期**（必填）。若实收 < 应收，状态变更为“部分收款”；若实收 > 应收，弹窗提示“实收金额大于应收金额，是否确认结清？”；若实收 = 应收，状态变更为“已结清”。
- **异常流定义**：
  - **多次录入逻辑**：部分收款场景下，每次录入实收金额采用**累加逻辑**（系统将本次录入金额与历史实收金额累加作为总实收金额）。
  - **超额收款拒绝结清**：若实收 > 应收且用户在弹窗中点击“取消”，收款状态保持“部分收款”，实收金额记录实际累加数值，差额标记为“待处理溢缴”。
- **验收标准 (AC)**：
  - AC1: 应收1000，第一次录入实收400，状态变为“部分收款”；第二次录入实收600，累加实收=1000，状态变为“已结清”。
  - AC2: 应收1000，录入实收1200，弹出确认框；点击“确认结清”，状态变为“已结清”，保留应收1000/实收1200记录，差额200标记“待处理溢缴”；点击“取消”，状态保持“部分收款”，实收记录为1200，差额200标记“待处理溢缴”。

**6. 应收账款仪表盘**
- **功能描述**：首页可视化展示“已收款、未到期、逾期”金额汇总与发票明细列表，支持按状态筛选。明细列表分页加载（每页20条），汇总数据定时聚合缓存（每5分钟），“确认回款”操作后触发实时刷新。
- **验收标准 (AC)**：
  - AC1: 确认回款后，刷新页面，仪表盘汇总金额与发票状态同步更新。
  - AC2: 无操作情况下，每隔5分钟刷新页面，汇总数据更新。

**7. 数据安全与隔离**
- **功能描述**：多租户数据严格隔离，接口层校验身份与归属权。财务敏感数据采用透明数据加密（TDE），**严禁对“金额”字段使用应用层加密**。

### P0.5-增强（可后置，缩短首次验证时间）

**1. 撤销回款确认机制**
- **功能描述**：确认回款后 168 小时内可撤销。撤销时，实收金额清零，收款状态恢复为“未收”，逾期分桶根据当前时间重算，记录状态变更日志。
- **异常流定义**：撤销回款确认对催款状态的影响——撤销时，若存在未发送的催款草稿则自动删除；若已有发送成功的催款记录，则保留发送记录但标记“该笔回款已被撤销”，催款状态与收款状态独立重算。
- **验收标准 (AC)**：
  - AC1: 撤销后，实收清零，状态变为“未收”，逾期天数按当前时间重算。
  - AC2: 撤销后，相关联的未发送草稿消失；已发送成功的催款记录仍保留，但带有“已撤销”标记。

### P1（体验增强与效率提升）

1. **AI 发票信息极速录入**：支持截图 OCR 识别或解析发票邮件自动提取信息。单字段识别准确率 ≥ 90%，置信度 <80% 高亮提示；识别失败自动清空，光标聚焦金额输入框，弹 Toast“识别失败，请手动输入”。
2. **多渠道催款触达（微信）**：待确认：需评估企业微信 API 合规性、成本及防封号策略后方可实施。

### P2（生态与商业化）

1. **第三方财务/支付工具打通**：支持数据同步与自动销账。
2. **多币种与跨国税务支持**：数据模型预留币种字段，MVP 仅支持人民币。

## 范围与非目标

**本期不做什么（防止范围发散）**：
- **不做重型会计与记账**：不涉及支出管理、资产负债表、利润表等标准会计功能。
- **不做项目与合同全流程管理**：不涉及提案、合同签署、任务进度追踪。
- **不做支付网关**：本期不提供客户直接在线支付的功能，仅做信息追踪与催款提醒。
- **不做完全自动发送催款**：为避免不可控的客情危机，本期催款动作仅停留在“生成草稿由用户确认后手动发送”。
- **不做海外场景支持**：MVP阶段仅聚焦本土场景，不支持多币种与跨国税务。

## 价值假设与待验证问题

**价值假设**：
1. 发票状态自动追踪与逾期预警，能让用户不再需要手动记忆和核对，从而获得更高的收款效率与更低的心理负担。
2. 自动化且分级的催款模板（生成草稿确认），能避免直接冲突的尴尬，让用户获得更顺畅的催款体验和更高的回款率。

**待验证问题（Open Questions）**：
- **多渠道催款合规与成本**：微信催款触达的 API 合规性、成本及防封号策略是什么？（决定 P1 微信催款是否能落地）
- **商业模式**：产品的商业模式是什么？是 SaaS 订阅、按催款成功次数抽成，还是基础免费+高级功能收费？（待确认：需在 MVP 验证后决策）

## 风险与依赖

1. **技术与合规风险**：
   - 邮件送达率风险：采用用户授权 SMTP/OAuth 可大幅降低被拦截风险，但授权流程复杂可能导致用户流失。
   - 微信/短信催款触达严重依赖第三方 API，存在被封号或接口限制的风险。
2. **用户信任风险**：
   - 催款涉及用户核心利益与客情关系，若系统误发催款信、话术不当或回复链路断裂，可能导致用户与客户关系破裂。
3. **指标度量风险**：
   - 核心价值指标（催款转化率）存在自选择偏差，MVP阶段无法建立严格因果归因，可能导致对产品价值的误判。
4. **跨团队依赖**：
   - **前置依赖**：P0 功能上线前，必须由法务团队审核分级催款话术的合规性与措辞风险。
   - **前置依赖**：需后端架构团队确认多租户隔离与 TDE 加密存储方案，满足财务类应用安全基线。

## ⑤ 终版 PRD

# 自由职业者智能催款与应收管理工具 PRD

**一句话定位**：专为独立自由职业者打造的轻量级“开票-追踪-催款”闭环工具，解决回款遗忘与催款尴尬，让每一笔收入落袋为安。

## 背景与问题

自由职业者（如独立设计师、开发者、撰稿人）在享受工作自由的同时，普遍面临严重的应收账款管理痛点：
1. **状态黑盒导致回款延迟**：项目交付后缺乏系统记录，发票状态散落在邮件和聊天记录中，经常遗忘哪些已开票、哪些未付款，导致资金回笼被动延迟。
2. **心理障碍导致不敢催款**：面对客户逾期，因碍于情面或不知如何措辞而迟迟不愿主动催款，严重影响个人现金流。
3. **对账繁琐导致效率低下**：月底年底对账时，需跨平台手动核对发票与打款记录，耗费大量时间且易出错。

**竞品现状与差异化机会**：
当前市场方案存在明显空白。Bonsai 偏重全流程且主攻海外；FreshBooks 和 Xero 属于重型会计软件，对独立工作者过于臃肿；Zoho Invoice 缺乏针对自由职业者心理的分级话术设计；Portmoneo 偏向支出端追踪，缺乏应收催收闭环。
本期产品将避开重型会计与全流程管理，专注于**“轻量级应收账款管理”**，并重点突破**“符合本土人情社会的智能催款”**，通过分级话术、草稿确认与定时发送机制，解决自由职业者“不敢催”的心理痛点。

## 目标用户与典型场景

**目标用户**：独立提供专业服务且需自行处理财务与收款的自由职业者（如独立设计师、开发者、撰稿人等）。MVP阶段优先聚焦国内本土自由职业者。

**典型场景**：
1. **开票与追踪**：项目完成并开具发票后，用户在工具中录入发票信息及客户联系方式并设定账期；系统自动记录发票状态，并在到期前提醒用户。
2. **逾期与催款**：发票逾期未付款，用户点击一键催款；工具根据逾期天数自动匹配分级话术生成催款邮件草稿，用户确认后可选择立即发送或设定定时发送（如2小时后自动发送，期间可反悔），发送后系统自动记录催款动作与时间。
3. **对账与决策**：用户收到线下打款后，在工具中手动确认回款；仪表盘清晰展示已付款、未付款、逾期发票的汇总数据与明细，辅助财务决策。

## 产品目标与成功指标

**产品目标**：
- **北极星指标**：缩短平均回款周期（从发票到期到实际回款的天数）。
  - *可控性假设*：是否回款本质由客户决定，但产品通过降低催款心理门槛和提供专业话术，能有效提高催款行为发生率和触达率，从而间接加速客户付款决策。
  - *退路指标*：若MVP证明回款周期不可控或归因困难，则将北极星退回为更前置但依然指向业务结果的代理指标——**“逾期发票首次催款时间缩短”**，而非仅停留在功能使用率的虚荣指标。
- **支撑性体验目标**：降低管理负担（代理指标：手动对账耗时减少）、降低催款心理门槛（代理指标：催款功能使用率与定时发送使用率）。

**成功指标（MVP验收门槛）**：
*注：以下为MVP验收节点的硬性阈值，基于行业常识与SaaS工具基准设定。*

1. **采用率**：注册首周内录入 ≥ 1张发票且完成一次状态推进（生成草稿或确认回款）的用户占比 ≥ 15%。
2. **功能渗透率**：产生逾期发票的用户中，使用“催款功能”的比例 ≥ 20%。
3. **核心价值验证（催款转化率）**：催款后7天回款转化率 = （成功发送催款后7天内确认回款的逾期发票数 / 成功发送催款的逾期发票总数）≥ 5%。
   - *归因说明*：由于MVP无法进行随机分流实验，该指标存在自选择偏差。MVP阶段仅作为**相关性验证指标**，后续通过同用户历史行为对比进行准实验评估。
4. **留存率**：核心用户的按月留存率（当月至少登录1次并产生有效操作的用户占比） ≥ 50%。*有效操作枚举：录入发票/生成草稿/发送催款/确认回款。*
5. **商业价值弱信号**：高级功能点击率 / 愿意付费调研评分（本期验证价值假设，监控付费意愿弱信号）。

## 核心功能与需求清单

### P0-核心（MVP 必需闭环）

**1. 发票与客户信息管理**
- **功能描述**：支持手动输入发票金额、账期、开票日期，及关联客户的基础信息（姓名、邮箱、手机号）。金额必须>0且最大值为 9999999999.99；邮箱需符合正则格式校验；账期不得早于开票日期；邮箱与姓名为必填，手机号为非必填。支持对未结清发票及客户信息编辑修改；支持逻辑删除发票（软删除，需二次确认）。
- **技术规范**：数据库强制使用 Decimal(12,2) 或将金额转换为“分”存储为 BigInt，严禁使用 Float/Double。时间字段存储 UTC，应用层根据用户配置的时区进行展示和计算（MVP默认 UTC+8）。
- **异常流与联动定义**：
  - **删除约束**：处于“发送中”的发票禁止删除；已有部分收款或结清记录的发票禁止删除（需先撤销回款确认）。
  - **编辑联动**：修改应收金额时，若该发票已有实收记录，需提示“修改应收金额将导致原回款确认被撤销，需重新核销”，用户确认后撤销回款状态并重算；修改账期时，逾期分桶实时重算，且已生成的未发送催款草稿自动失效删除。
- **验收标准 (AC)**：
  - AC1: 输入合法数据，保存成功，列表可见新发票。
  - AC2: 输入金额<=0、非法邮箱、账期早于开票日，对应字段下方展示红色错误提示，无法提交。
  - AC3: 删除未结清且无发送中/回款记录的发票时弹出二次确认弹窗，确认后列表不显示，仪表盘统计不包含此发票。
  - AC4: 处于“发送中”的发票，删除按钮置灰且 Tooltip 提示“发送中的发票不可删除”。
  - AC5: 修改未结清发票的账期后，逾期分桶重算，原未发送草稿被清除。

**2. 发票状态追踪与到期预警**
- **功能描述**：采用两个正交维度表达状态。收款状态：未收、已结清；逾期分桶（派生值）：未到期、逾期1-6天、逾期7-29天、逾期30天+。弃用“读时计算+写回缓存”，改为基于事件驱动或定时任务（Cron Job，每小时跑批一次）更新物理状态位。到期前1天及逾期首日站内信提醒；逾期分桶跃迁时再次触发站内信提醒。
- **异常流定义**：长期未登录场景下，站内信仅触发当前所处分桶的跃迁提醒，历史未触发的分桶提醒不补发。
- **验收标准 (AC)**：
  - AC1: 跨过到期日且跑批执行后，发票逾期分桶更新为“逾期1-6天”，并收到站内信。
  - AC2: 用户逾期第1天未登录，第10天首次登录，系统仅展示“逾期7-29天”的跃迁站内信，不补发“逾期1-6天”的提醒。

**3. 分级催款话术与草稿生成（邮件）**
- **功能描述**：话术分三级：温和提醒（1-6天）、正式催款（7-29天）、最终通知（30天+）。一键生成对应级别邮件草稿，重复生成将覆盖旧草稿（覆盖前二次确认）。MVP砍掉短信兜底通道，仅支持邮件催款。若客户无邮箱（极端异常数据），提示用户补充邮箱。
- **前置依赖**：话术模板需经法务团队审核通过后方可上线。
- **验收标准 (AC)**：
  - AC1: 逾期1-6天生成草稿，话术为“温和提醒”；修改系统时间至逾期10天，生成草稿话术为“正式催款”。
  - AC2: 覆盖已存在的草稿时，弹出“覆盖确认”弹窗；取消则保留旧草稿。
  - AC3: 客户信息缺失邮箱时，点击生成草稿提示“请先补充客户邮箱”。

**4. 邮件发送与状态追踪**
- **功能描述**：MVP采用系统代发邮件（如 noreply@product.com 代发，Reply-To 设为用户邮箱），以降低用户配置门槛，保障采用率。用户确认草稿后可选择“立即发送”或“定时发送”（默认2小时后发送，期间可手动取消）。状态枚举：待发送、发送中、发送成功、发送失败、已退信。退信时通过站内信预警。发送中禁止生成新草稿。
- **异常流定义**：
  - **异步状态对齐**：引入异步任务队列管理发送生命周期，对接网关 Webhook 异步更新状态。若因网络延迟导致系统超时置为“发送失败”，但后续收到网关“发送成功”或“已退信”回调，系统应自动修正状态，并通过站内信提示用户“上一次催款邮件实际已送达/退信，请勿重复发送”。
- **验收标准 (AC)**：
  - AC1: 邮件发送成功，状态变为“发送成功”；模拟网关返回退信 Webhook，状态变为“已退信”并收到站内信。
  - AC2: 邮件处于“发送中”时，点击“生成催款草稿”按钮，提示“上一次发送尚未完成，请稍后”。
  - AC3: 设定定时发送后，在倒计时内可点击“取消发送”，状态退回为“待发送”。
  - AC4: 模拟超时置为失败后，触发真实成功回调，状态自动修正为“发送成功”，并收到防重复催款的站内信提示。

**5. 手动确认回款/销账**
- **功能描述**：为坚守轻量级底线，MVP砍掉部分收款累加与溢缴逻辑。发票状态仅保留“未收”和“已结清”。用户录入实收金额与实际到账日期（必填）。若实收 ≠ 应收，弹窗提示“实收与应收金额不符，是否确认按实收金额结清并自动调整应收金额？”；若确认，则应收金额被修改为实收金额，状态变为“已结清”；若取消，则返回修改。
- **验收标准 (AC)**：
  - AC1: 应收1000，录入实收1000，状态变为“已结清”。
  - AC2: 应收1000，录入实收800，弹出确认框；点击“确认结清”，应收金额变更为800，状态变为“已结清”；点击“取消”，留在当前页面，实收金额可继续编辑。

**6. 应收账款仪表盘**
- **功能描述**：首页可视化展示“已收款、未到期、逾期”金额汇总与发票明细列表，支持按状态筛选。明细列表分页加载（每页20条），汇总数据定时聚合缓存（每5分钟）。所有导致金额与状态变更的操作（新增、删除、编辑、确认回款）均触发仪表盘实时刷新。
- **验收标准 (AC)**：
  - AC1: 确认回款、新增发票、删除发票后，刷新页面，仪表盘汇总金额与发票状态同步更新。
  - AC2: 无操作情况下，每隔5分钟刷新页面，汇总数据更新。

**7. 数据安全与隔离**
- **功能描述**：多租户数据严格隔离，接口层校验身份与归属权。财务敏感数据采用透明数据加密（TDE），严禁对“金额”字段使用应用层加密。

### P0.5-增强（可后置，缩短首次验证时间）

**1. 撤销回款确认机制**
- **功能描述**：确认回款后 168 小时内可撤销。撤销时，实收金额清零，收款状态恢复为“未收”，应收金额恢复为原始开票金额，逾期分桶根据当前时间重算，记录状态变更日志。
- **异常流定义**：撤销时，若存在未发送的催款草稿则自动删除；若已有发送成功的催款记录，则保留发送记录但标记“该笔回款已被撤销”，催款状态与收款状态独立重算。
- **验收标准 (AC)**：
  - AC1: 撤销后，实收清零，应收恢复原值，状态变为“未收”，逾期天数按当前时间重算。
  - AC2: 撤销后，相关联的未发送草稿消失；已发送成功的催款记录仍保留，但带有“已撤销”标记。

### P1（体验增强与效率提升）

1. **AI 发票信息极速录入**：支持截图 OCR 识别或解析发票邮件自动提取信息。提供标准测试发票图片集（含清晰、模糊、遮挡等分类）作为验收基线；置信度 <80% 高亮提示；关键字段缺失或识别失败自动清空，光标聚焦金额输入框，弹 Toast“识别失败，请手动输入”。AC改为确定性断言：标准测试集中指定模糊图片必须高亮提示，指定遮挡图片必须清空并弹窗。
2. **多渠道催款触达（短信与微信）**：待确认：需评估企业微信 API 合规性、成本及防封号策略；短信需解决实名审核与模板报备问题。依赖 P0-1 中补充的手机号字段。
3. **个性化发信（SMTP/OAuth）**：支持用户授权绑定个人邮箱发信，解决代发邮件可能被拦截或进入垃圾箱的问题。需预调研 Gmail/Outlook 等主流服务商 OAuth2 接入限制，并设计严密的 Token 加密存储方案。

### P2（生态与商业化）

1. **第三方财务/支付工具打通**：支持数据同步与自动销账。
2. **多币种与跨国税务支持**：数据模型预留币种字段，MVP 仅支持人民币。

## 范围与非目标

**本期不做什么（防止范围发散）**：
- **不做重型会计与记账**：不涉及支出管理、资产负债表、利润表等标准会计功能，不处理部分收款、溢缴款与红字冲销。
- **不做项目与合同全流程管理**：不涉及提案、合同签署、任务进度追踪。
- **不做支付网关**：本期不提供客户直接在线支付的功能，仅做信息追踪与催款提醒。
- **不做完全自动发送催款**：为避免不可控的客情危机，本期催款动作必须有用户主动确认（立即发送或定时发送），禁止无人工干预的自动发送。
- **不做海外场景支持**：MVP阶段仅聚焦本土场景，不支持多币种与跨国税务。

## 价值假设与待验证问题

**价值假设**：
1. 发票状态自动追踪与逾期预警，能让用户不再需要手动记忆和核对，从而获得更高的收款效率与更低的心理负担。
2. 自动化且分级的催款模板配合定时发送机制，能降低专业措辞门槛与发送瞬间的心理阻力（缓冲期提供反悔余地），让用户获得更顺畅的催款体验和更高的回款率。

**待验证问题（Open Questions）**：
- **多渠道催款合规与成本**：微信/短信催款触达的 API 合规性、成本及防封号策略是什么？（决定 P1 微信/短信催款是否能落地）
- **商业模式**：产品的商业模式是什么？是 SaaS 订阅、按催款成功次数抽成，还是基础免费+高级功能收费？（待确认：需在 MVP 验证后决策）

## 风险与依赖

1. **技术与合规风险**：
   - 邮件送达率风险：系统代发邮件可能被识别为垃圾邮件，需配置好 SPF/DKIM，P1 阶段通过 SMTP/OAuth 彻底解决。
   - 微信/短信催款触达严重依赖第三方 API，存在被封号或接口限制的风险。
2. **用户信任风险**：
   - 催款涉及用户核心利益与客情关系，若系统误发催款信、话术不当或回复链路断裂，可能导致用户与客户关系破裂。
3. **指标度量风险**：
   - 核心价值指标（催款转化率）存在自选择偏差，MVP阶段无法建立严格因果归因，可能导致对产品价值的误判。
4. **跨团队依赖**：
   - **前置依赖**：P0 功能上线前，必须由法务团队审核分级催款话术的合规性与措辞风险。
   - **前置依赖**：需后端架构团队确认多租户隔离与 TDE 加密存储方案，满足财务类应用安全基线。
